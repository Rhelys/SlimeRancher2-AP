## IL2CPP vs Mono — Critical Differences

This is **BepInEx 6 IL2CPP**. Many patterns from BepInEx 5 (Mono) tutorials do not apply.

| Concern | Mono (BepInEx 5) | IL2CPP (BepInEx 6) — THIS PROJECT |
|---|---|---|
| Entry point | `BaseUnityPlugin.Awake()` | `BasePlugin.Load()` |
| Update loop | Built into `BaseUnityPlugin` | Must inject via `AddComponent<ApUpdateBehaviour>()` |
| Custom MonoBehaviours | Just define the class | Must call `ClassInjector.RegisterTypeInIl2Cpp<T>()` first |
| MonoBehaviour constructors | Default `()` ctor | **Must** have `(IntPtr handle) : base(handle) { }` |
| Game type DLLs | Original managed DLLs | Generated interop DLLs in `BepInEx/interop/` |
| Interface casting | Implicit | Must use `.Cast<IFoo>()` from Il2CppInterop |
| `GetComponent<T>()` | Works for any class | **Only works for MonoBehaviour subclasses** |
| Non-MonoBehaviour game objects | Rare | Common — e.g. `ActorUpgradeHandler` extends `Il2CppSystem.Object` |
| Legacy Input | `UnityEngine.Input` | SR2 uses **New Input System**: `UnityEngine.InputSystem.Keyboard.current` |
| GUI scrollviews | `GUI.BeginScrollView` works | `GUI.BeginScrollView` **is stripped** in IL2CPP builds — do not use |
| `GUI.Window` / function delegates | Works | `GUI.WindowFunction` delegate creation **fails** — draw directly in `OnGUI` |

### IL2CPP-Specific Gotchas Encountered

- **`GetComponent<ActorUpgradeHandler>()` always returns null** — `ActorUpgradeHandler` extends `Il2CppSystem.Object`, not `MonoBehaviour`. It is constructed with `new ActorUpgradeHandler(ActorModel parent)`. Access it by patching its constructor (see `ActorUpgradeHandlerPatch.cs`).
- **`CurrencyDefinition` is not implicitly `ICurrency`** — must use `.Cast<ICurrency>()`.
- **`OnGUI` fires multiple times per frame** (layout + repaint passes) — keyboard input like `wasPressedThisFrame` must only be read in `Update()`, never in `OnGUI()`, or keys will double-fire.
- **Cursor stays locked in debug panel** — game manages `Cursor.lockState`; save and restore it manually on panel open/close.
- **`System.Random` vs `UnityEngine.Random` ambiguity** — qualify as `System.Random` explicitly when both are in scope.
- **`using` directives needed for interop types** — game types live in namespaces like `Il2CppMonomiPark.SlimeRancher.Economy`; always check the `apis/` decompiled file for the correct namespace.

## Architecture Overview

```
Plugin.cs                   ← BepInEx 6 entry point (BasePlugin.Load)
ApUpdateBehaviour           ← Injected MonoBehaviour for per-frame queue processing
```

### Data Flow

1. **Connection**: `ConnectionUI` → `ArchipelagoClient.Connect()` (off-thread, `Task.Run`) → `TryConnectAndLogin` → `OnConnected`
2. **Items received**: AP server → `session.Items.ItemReceived` (network thread) → `_itemQueue` → `ApUpdateBehaviour.Update()` → `ItemHandler.Apply()` (main thread only)
3. **Location checks**: Game event → Harmony Postfix patch → `ArchipelagoClient.SendCheck(id)` → `session.Locations.CompleteLocationChecks()`
4. **DeathLink**: Player death → `PlayerDeathPatch` → `DeathLinkHandler.SendDeath()` | Received death → `_pendingDeaths` queue → `ProcessDeathQueue()` (main thread) → `KillPlayer()`
5. **Save state**: `ApSaveManager` writes `BepInEx ConfigFile` keyed by `{seed}_{slotName}` — persists checked locations and item index high-water mark

### Reconnect / Item Deduplication

`session.Items.ItemReceived` replays **all historical items** on reconnect (due to `ItemsHandlingFlags.AllItems`). Items with `item.Index <= ApSaveManager.LastItemIndex` are skipped. `LastItemIndex` is updated and persisted after each successful grant.

## Confirmed Game API Details

These have been verified against the decompiled `apis/` files and/or in-game log dumps. Do not guess — always check `apis/` before writing game API calls.

### Key Classes and Access Paths

| What | How to access |
|---|---|
| Player GameObject | `SceneContext.Instance.Player` (type: `GameObject`) |
| Player state / currency / health | `SceneContext.Instance.PlayerState` (type: `PlayerState`) |
| Player data model | `SceneContext.Instance.PlayerState._model` (type: `PlayerModel`) |
| Gadget director / refinery | `SceneContext.Instance.GadgetDirector` |

### ActorUpgradeHandler

- **Not a MonoBehaviour** — `GetComponent<ActorUpgradeHandler>()` always returns null
- Extends `Il2CppSystem.Object`; constructor signature: `ActorUpgradeHandler(ActorModel parent)` (CallerCount = 1 — instantiated once per game session)
- Access pattern: Harmony Postfix on the constructor caches it into `ItemHandler.UpgradeHandler`
- Grant method: `upgradeHandler.ApplyUpgrade(UpgradeDefinition definition, int level)`

### Death / DeathLink

- Kill the player: `player.GetComponent<PlayerDeathHandler>().OnDeath(null, null, "DeathLink")`
  - Fallback if component missing: `SceneContext.Instance.PlayerState.SetHealth(0)` (does NOT trigger full death pipeline)
- Detect player death for sending: Harmony Prefix on `PlayerDeathHandler.OnDeath`
- DeathLink feedback guard: `_isSendingDeath` flag in `DeathLinkHandler` prevents re-broadcasting a death we caused

### Region Gates

- Gate method: `WorldStatePrimarySwitch.SetStateForAll(SwitchHandler.State.DOWN, true)`
  - Note: method is `SetStateForAll`, NOT `SetState` (which does not exist)
- Gate patch: Prefix on `SetStateForAll` returns false (blocks original) unless `ApSaveManager.IsRegionUnlocked(switchName)`
- Find switches: `Resources.FindObjectsOfTypeAll<WorldStatePrimarySwitch>()`

### Harmony Patch Rules

- Parameter names in Postfix/Prefix must **exactly** match the original method's parameter names — use `nameof()` or verify from `apis/` decompile
- Example past bug: `FabricatorPatch` used `GadgetDefinition gadgetDef` but actual name is `blueprint` — caused "Parameter not found" error at load

### Map Nodes

- Patch: `MapNodeActivator.Activate()` Postfix (confirmed class name from decompile)

`session.SetGoalAchieved()`. All detectors live in `Archipelago/GoalHandler.cs`.

## Drone Patches (mod side)

### Research Drones (~23, narrative/lore drones)
- **What**: Gigi's journal drones; player approaches to trigger a dialogue log
- **Check sent when**: Player first activates a Research Drone
- **Patch class**: `ResearchDroneActivator` (`MonomiPark.SlimeRancher.World.ResearchDrone` namespace, in `Assembly-CSharp.dll`)
- **Patch method**: `ResearchDroneActivator.OnInteract()` — Postfix
- **Get drone identity from**: `__instance._researchDroneController.ResearchDroneEntry` (a `ResearchDroneEntry` asset with a stable name/ID)
- **Dedup**: `ResearchDroneController._model` tracks whether it has been activated — check `_state == DroneState.ACTIVATED` or model already activated before sending check
- **Lookup**: `ResearchDroneEntry` asset name → `LocationTable`
- **Location IDs**: 819450–819479
- **Using**: `Il2CppMonomiPark.SlimeRancher.World.ResearchDrone`

### Ghostly Drones (10, collectible drones)
- **What**: Flying mechanical drones (`ComponentAcqDrone` type) that give a Drone Station Module when interacted with
- **Check sent when**: Player successfully interacts with the drone via `ComponentAcquisitionDroneUIInteractable.OnInteract()`
- **Patch class**: `ComponentAcquisitionDroneUIInteractable` (`MonomiPark.SlimeRancher.UI` namespace, in `Assembly-CSharp.dll`)
- **Patch method**: `ComponentAcquisitionDroneUIInteractable.OnInteract()` — Postfix
- **Get drone identity from**: `__instance._componentAcqDrone.gameObject.name` — match against `LocationTable`
- **Lookup**: `gameObject.name` → `LocationTable` (Ghostly Drone section)
- **Location IDs**: 819480–819494
- **Note**: Drone Station Modules can also be bought at ProntoMart — patching `OnInteract` on the world drone specifically avoids the purchase path
- **Using**: `Il2CppMonomiPark.SlimeRancher.UI`, `Il2CppMonomiPark.SlimeRancher.Drone`

### Shadow Plort Doors (25, Grey Labyrinth)
- **What**: Doors in the Grey Labyrinth opened by depositing Shadow Plorts
- **Check sent when**: Door fully fills and opens
- **Patch class**: `PlortDepositor` (root namespace, in `Assembly-CSharp.dll`)
- **Patch method**: `PlortDepositor.ActivateOnFill()` — Postfix (fires exactly once when fill completes and the door activates)
- **Alternative**: `PlortDepositor.OnFilledChangedFromModel()` — also fires on fill, but `ActivateOnFill` is more precise (CallerCount = 1, only called on completion)
- **Get door identity from**: `__instance.gameObject.name` — match against `LocationTable`
- **Lookup**: `gameObject.name` → `LocationTable` (Shadow Plort Door section)
- **Location IDs**: 819200–819249
- **Note**: `PlortDepositor` is also used for non-Labyrinth plort locks (e.g. Gordo reward doors) — filter by checking `__instance._catchIdentifiableType.name == "ShadowPlort"` to only trigger on the correct door type

## How to Decompile Game APIs

Use `ilspycmd` to decompile `Assembly-CSharp.dll` into per-type `.cs` files in the `apis/` folder:

```bash
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -p -o apis/
```

The `apis/` folder is excluded from compilation (`<Compile Remove="apis\**" />`) and from git (`.gitignore`). Always search `apis/` to confirm class names, method signatures, and parameter names before writing any patch or game API call.

### ilspycmd — Decompiling a Single Type

To decompile one specific class without regenerating the whole `apis/` folder:

```bash
# List all classes matching a keyword to find the correct fully-qualified IL2CPP name:
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -l c 2>&1 | grep -v "not using\|Latest version" | grep -i "UpgradeComponent"

# Decompile a single type by its fully-qualified IL2CPP name:
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -t "Il2CppMonomiPark.SlimeRancher.Player.UpgradeComponentsModel" 2>&1 | grep -v "not using\|Latest version"
```

**Critical rules:**
- **Use forward slashes** in the DLL path — backslashes cause shell parse errors in bash.
- **The `-t` name must use the IL2CPP interop prefix** — type names in the interop DLL have `Il2Cpp` prepended to the original C# namespace (e.g. `MonomiPark.SlimeRancher.Player.Foo` → `Il2CppMonomiPark.SlimeRancher.Player.Foo`). Using the original C# namespace gives `Could not find type definition`.
- **Always pipe through `grep -v "not using\|Latest version"`** — ilspycmd prints a version-update warning to stdout on every invocation; without filtering it pollutes the output.
- **Use `-l c` to discover names** before decompiling — if you're unsure of the exact namespace, list all classes and grep for the class name first.
