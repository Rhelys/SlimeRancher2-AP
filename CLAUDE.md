## IL2CPP vs Mono — Critical Differences

This is **BepInEx 6 IL2CPP**. Many patterns from BepInEx 5 (Mono) tutorials do not apply.

| Concern | Mono (BepInEx 5) | IL2CPP (BepInEx 6) — THIS PROJECT |
|---|---|---|
| Entry point | `BaseUnityPlugin.Awake()` | `BasePlugin.Load()` |
| Update loop | Built into `BaseUnityPlugin` | Must inject via `AddComponent<ApUpdateBehaviour>()` |
| Custom MonoBehaviours | Just define the class | Must call `ClassInjector.RegisterTypeInIl2Cpp<T>()` first |
| MonoBehaviour constructors | Default `()` ctor | **Must** have `(IntPtr handle) : base(handle) { }` |
| Game type DLLs | Original managed DLLs | Generated interop DLLs in `BepInEx/interop/` |
| Interface casting | Implicit | Must use `.Cast<IFoo>()` from Il2CppInterop |
| `GetComponent<T>()` | Works for any class | **Only works for MonoBehaviour subclasses** |
| Non-MonoBehaviour game objects | Rare | Common — e.g. `ActorUpgradeHandler` extends `Il2CppSystem.Object` |
| Legacy Input | `UnityEngine.Input` | SR2 uses **New Input System**: `UnityEngine.InputSystem.Keyboard.current` |
| GUI scrollviews | `GUI.BeginScrollView` works | `GUI.BeginScrollView` **is stripped** in IL2CPP builds — do not use |
| `GUI.Window` / function delegates | Works | `GUI.WindowFunction` delegate creation **fails** — draw directly in `OnGUI` |

### IL2CPP-Specific Gotchas Encountered

- **`GetComponent<ActorUpgradeHandler>()` always returns null** — `ActorUpgradeHandler` extends `Il2CppSystem.Object`, not `MonoBehaviour`. It is constructed with `new ActorUpgradeHandler(ActorModel parent)`. Access it by patching its constructor (see `ActorUpgradeHandlerPatch.cs`).
- **`CurrencyDefinition` is not implicitly `ICurrency`** — must use `.Cast<ICurrency>()`.
- **`OnGUI` fires multiple times per frame** (layout + repaint passes) — keyboard input like `wasPressedThisFrame` must only be read in `Update()`, never in `OnGUI()`, or keys will double-fire.
- **Cursor stays locked in debug panel** — game manages `Cursor.lockState`; save and restore it manually on panel open/close.
- **`System.Random` vs `UnityEngine.Random` ambiguity** — qualify as `System.Random` explicitly when both are in scope.
- **`using` directives needed for interop types** — game types live in namespaces like `Il2CppMonomiPark.SlimeRancher.Economy`; always check the `apis/` decompiled file for the correct namespace.

## Architecture Overview

```
Plugin.cs                   ← BepInEx 6 entry point (BasePlugin.Load)
ApUpdateBehaviour           ← Injected MonoBehaviour for per-frame queue processing
```

### Data Flow

1. **Connection**: `ConnectionUI` → `ArchipelagoClient.Connect()` (off-thread, `Task.Run`) → `TryConnectAndLogin` → `OnConnected`
2. **Items received**: AP server → `session.Items.ItemReceived` (network thread) → `_itemQueue` → `ApUpdateBehaviour.Update()` → `ItemHandler.Apply()` (main thread only)
3. **Location checks**: Game event → Harmony Postfix patch → `ArchipelagoClient.SendCheck(id)` → `session.Locations.CompleteLocationChecks()`
4. **DeathLink**: Player death → `PlayerDeathPatch` → `DeathLinkHandler.SendDeath()` | Received death → `_pendingDeaths` queue → `ProcessDeathQueue()` (main thread) → `KillPlayer()`
5. **Save state**: `ApSaveManager` writes `BepInEx ConfigFile` keyed by `{seed}_{slotName}` — persists checked locations and item index high-water mark

### Reconnect / Item Deduplication

`session.Items.ItemReceived` replays **all historical items** on reconnect (due to `ItemsHandlingFlags.AllItems`). Items with `item.Index <= ApSaveManager.LastItemIndex` are skipped. `LastItemIndex` is updated and persisted after each successful grant.

### Death / DeathLink

- Kill the player: `player.GetComponent<PlayerDeathHandler>().OnDeath(null, null, "DeathLink")`
  - Fallback if component missing: `SceneContext.Instance.PlayerState.SetHealth(0)` (does NOT trigger full death pipeline)
- Detect player death for sending: Harmony Prefix on `PlayerDeathHandler.OnDeath`
- DeathLink feedback guard: `_isSendingDeath` flag in `DeathLinkHandler` prevents re-broadcasting a death we caused

### Region Gates

- EV / SS gates: `WorldStatePrimarySwitch` buttons in `zoneFields` (`ruinSwitch` / `ruinSwitch (2)`).
  Patched via **`RegionGateActivatePatch`** — Prefix on `WorldStatePrimarySwitch.Activate()`
  blocks the press (and sends the location check) while the region is locked, keeping
  `_switchEnabled` untouched so the button stays pressable. The old `SetStateForAll` patch was
  removed — its native prologue changed in the 5/13/2026 game update and crashed the trampoline.
- PB gate: a `PuzzleSlotLockable` plort door (`RegionTable.PBGatePosKey`), NOT a
  `WorldStatePrimarySwitch`. Opened from native code, so no Harmony patch fires —
  **`PlortDoorPoller`** polls `ShouldUnlock()` from the Update loop; unauthorized entry is
  reverted by **`GateReturnEnforcer`**.
- Labyrinth entrances: beam puzzles → `WorldStateInvisibleSwitch.SetStateForAll` —
  observed via **`InvisibleSwitchPatch`** (safe to patch; fires repeatedly while a beam is active).
- Find switches: `Resources.FindObjectsOfTypeAll<WorldStatePrimarySwitch>()`

### Harmony Patch Rules

- Parameter names in Postfix/Prefix must **exactly** match the original method's parameter names — use `nameof()` or verify from `apis/` decompile
- Example past bug: `FabricatorPatch` used `GadgetDefinition gadgetDef` but actual name is `blueprint` — caused "Parameter not found" error at load

### RadiantSlimeDirector (confirmed from fresh 5/13/2026 decompile)

- **Namespace**: `Il2CppMonomiPark.SlimeRancher.Slime` (not root namespace — was incorrectly thought absent from interop; it IS present, ilspycmd just failed to write its file during stale bulk decompile)
- **`DrawFromRadiantShuffleBag` signature** (changed in 5/13 update): `public bool DrawFromRadiantShuffleBag(IdentifiableType id, SceneGroup sceneGroup)` — added `SceneGroup sceneGroup` second parameter. Old patch that omitted this parameter crashed the trampoline.
- **`RadiantSlimesModel` property getter**: `CallerCount(0)`, no CachedScanResults — Cpp2IL found 0 managed callers. Avoid calling this property; the backing field `_radiantSlimesModel` is a safer direct field access if needed.
- **`DEBUG_ForceRadiantSpawn`**: direct bool field access (not a method call) — safe to set, but native `DrawFromRadiantShuffleBag` code does NOT check it (confirmed: has no effect on spawning in shipping builds).
- **`DEBUG_BagSizeScalar`**: direct float field access — same caveat, not read by shipping native code.
- **`ResizeBagsIfNeeded()`**: `CallerCount(2)` — valid, called by game. Can be invoked safely to force bag resize after config changes.
- **Patching `DrawFromRadiantShuffleBag`**: now safe with correct signature `(IdentifiableType id, SceneGroup sceneGroup)`. Use `[HarmonyPatch(typeof(RadiantSlimeDirector), "DrawFromRadiantShuffleBag")]` with Postfix `(ref bool __result, IdentifiableType id, SceneGroup sceneGroup)`.

## How to Decompile Game APIs

> **⚠️ MANDATORY RULE: Always use per-type decompile (`-t`) for any API lookup.**
> The bulk decompile (`-p -o apis/`) is **not reliable** — ilspycmd silently skips writing some type files with no error output. In the 5/13/2026 update, `RadiantSlimeDirector.cs` was never written during a full bulk run, leading to wrong assumptions about its namespace and method signatures that caused runtime crashes. **Never trust the `apis/` folder as the source of truth for any specific type.** Always run a fresh `-t` decompile for the exact type you are working with.

### Standard Workflow: Always Decompile the Target Type First

Before writing or modifying **any** patch or game API call, run a fresh single-type decompile for every class you will touch:

```bash
# Step 1 — Find the exact IL2CPP type name (if unsure of namespace):
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -l c 2>&1 | grep -v "not using\|Latest version" | grep -i "MyClassName"

# Step 2 — Decompile that specific type fresh from the current interop DLL:
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -t "Il2CppMonomiPark.SlimeRancher.MyNamespace.MyClassName" 2>&1 | grep -v "not using\|Latest version"
```

This is fast (under 1 second per type) and guaranteed to reflect the current game version. Do it for **every type** you patch or call into — even if it's in `apis/` already, even if it looks familiar, even if nothing appears to have changed.

### Bulk Decompile (rarely needed)

The full bulk decompile regenerates the entire `apis/` folder and is only useful for initial exploration across many types at once. It is **not a substitute** for per-type decompile when writing code.

```bash
ilspycmd "E:/SteamLibrary/steamapps/common/Slime Rancher 2/BepInEx/interop/Assembly-CSharp.dll" -p -o apis/
```

The `apis/` folder is excluded from compilation (`<Compile Remove="apis\**" />`) and from git (`.gitignore`).

> **⚠️ CRITICAL: The `apis/` folder goes stale after game updates AND may be incomplete even when fresh.**
> The interop DLLs in `BepInEx/interop/` are regenerated by BepInEx on every game launch after an update, but `apis/` must be manually regenerated separately. Even a freshly-run bulk decompile can silently omit files. **Always use `-t` per-type decompile as your primary source; treat `apis/` as a secondary convenience index only.**
> Stale/incomplete `apis/` data has caused wrong patch decisions (e.g. missing method parameters, incorrect namespace assumptions, incorrect "class absent" conclusions).

### ilspycmd Rules

**Critical rules:**
- **Use forward slashes** in the DLL path — backslashes cause shell parse errors in bash.
- **The `-t` name must use the IL2CPP interop prefix** — type names in the interop DLL have `Il2Cpp` prepended to the original C# namespace (e.g. `MonomiPark.SlimeRancher.Player.Foo` → `Il2CppMonomiPark.SlimeRancher.Player.Foo`). Using the original C# namespace gives `Could not find type definition`.
- **Always pipe through `grep -v "not using\|Latest version"`** — ilspycmd prints a version-update warning to stdout on every invocation; without filtering it pollutes the output.
- **Use `-l c` to discover names** before decompiling — if you're unsure of the exact namespace, list all classes and grep for the class name first.
- **A missing file in `apis/` does NOT mean the class doesn't exist** — run `-l c | grep` to check before concluding a class is absent from the interop DLL.
